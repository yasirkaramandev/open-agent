# Security review

Reviewed: 2026-07-24

## Threat model

OpenCalc Studio is a static, client-side calculator. Expressions are processed
locally by the tokenizer, parser, and evaluator; there is no backend, account,
credential, telemetry, advertising, or analytics integration.

The only persistent browser data is `localStorage`:

- `opencalc.settings` stores calculator and theme preferences.
- `opencalc.history` stores expressions and their formatted results unless
  private mode is enabled.

Memory-register state is held only in React memory. The PWA service worker
precaches the same-origin application shell for offline use.

The relevant threats are malicious clipboard text, tampered `localStorage`,
script or markup injection, compromised or vulnerable build dependencies, and
an over-broad service-worker cache or network scope. Browser storage is not a
security boundary: another script executing on the same origin could read or
change it, so calculator history must not be treated as confidential.

## Review and hardening performed

- Searched application HTML, configuration, public assets, and `src` for
  `eval(...)`, `new Function`, `dangerouslySetInnerHTML`, remote script sources,
  dynamic remote imports, analytics identifiers, and browser networking APIs.
  No application use was found. The calculator uses the pure
  `tokenize -> parse -> evaluate` pipeline and never performs JavaScript code
  generation.
- Confirmed the only page script is a same-origin module generated by Vite.
  There are no remote `<script>` elements or third-party analytics.
- Moved the theme `color-scheme` declaration from a DOM style mutation into
  the self-hosted stylesheet, so the application needs no inline scripts or
  styles under CSP.
- Confirmed there are no `fetch`, `XMLHttpRequest`, `WebSocket`, `EventSource`,
  or beacon calls in application source. The generated service worker performs
  only the same-origin requests required to precache and serve its listed local
  assets.
- Added a fail-closed paste handler. Clipboard text is limited to 256
  characters and 256 tokens, accepts only ASCII numbers, arithmetic operators,
  parentheses, and whitespace, and is reconstructed from the existing
  tokenizer's tokens before being sent to `compute`. Unsupported text, such as
  identifiers or markup, is rejected and is never passed raw to the engine.
- Reviewed storage paths. Stored settings and history are schema-checked before
  use; invalid records are removed. History is rendered as React text, and a
  recalled expression still passes through the calculator tokenizer/parser.
- Confirmed the generated service worker scope is the application origin and
  its precache manifest contains only relative, same-origin application assets.
- Exercised the production preview in a browser. The service worker reached the
  `activated` state, the calculator remained functional, no CSP or console
  errors were emitted, and no cross-origin request was observed.

React's production runtime contains internal support code and property-name
strings for `dangerouslySetInnerHTML`; the application does not call or pass
that prop. Likewise, dependency documentation URL strings in a bundle are not
remote script loads or runtime requests.

## Content Security Policy

`index.html` contains this policy:

```text
default-src 'none';
base-uri 'none';
script-src 'self';
style-src 'self';
img-src 'self';
connect-src 'self';
font-src 'self';
manifest-src 'self';
worker-src 'self';
object-src 'none';
frame-src 'none';
media-src 'self';
form-action 'none'
```

The policy has no `unsafe-inline`, `unsafe-eval`, `data:`, `blob:`, or
third-party source allowance. Scripts, styles, images, connections, fonts,
the manifest, and workers are restricted to the application origin.
`object-src 'none'` blocks `<object>` and `<embed>` content; `frame-src 'none'`
and `form-action 'none'` remove two additional exfiltration paths.
`worker-src 'self'` is the allowance needed by the same-origin PWA service
worker.

The policy is intentionally delivered by `<meta>` as requested. A production
host should also send the same policy as an HTTP response header if available;
header delivery can additionally enforce directives such as
`frame-ancestors`, which are not supported in a meta policy.

## Dependency audit

The initial full `npm audit` reported five development-tool findings: three
moderate, one high, and one critical. Production dependencies already had zero
findings. The actionable Vite, esbuild, and Vitest advisories were resolved by
upgrading the compatible toolchain:

- Vite `6.4.3`
- esbuild `0.25.12` (transitive through Vite)
- Vitest `3.2.7`
- `@vitejs/plugin-react` `4.7.0`

Final results:

- `npm audit`: 0 vulnerabilities across 490 dependencies.
- `npm audit --omit=dev`: 0 production vulnerabilities.
- Accepted unfixable or dev-only advisories: none.

## Verification

- Security grep over application-owned source/config/HTML: no `eval(...)`,
  `new Function`, `dangerouslySetInnerHTML`, or remote HTTP(S) script source.
- `npm test`: 49 passed.
- `npm run typecheck`: passed with no diagnostics.
- `npm run build`: passed; PWA precached 13 same-origin entries.
- `npm run test:e2e`: 18 passed across Chromium, Firefox, and WebKit.
- Production preview browser check: calculator and service worker operational
  under CSP, with zero cross-origin requests and zero console errors.
- `npm audit`: 0 vulnerabilities.
